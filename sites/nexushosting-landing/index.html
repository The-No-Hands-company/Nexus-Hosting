The idea

Your site doesn't live on one server.

Every hosting company today is a single point of control. They can take your site down, throttle it, change pricing, or go out of business.

Nexus Hosting is different. When you deploy, your site replicates automatically to every node in the network. Each node is run by an independent operator — a person, not a company. No one entity controls the network.

Nodes find each other, verify identity with Ed25519 cryptographic signatures, and sync sites automatically on every deploy. If a node goes offline, the others keep your site running.

Right now the network has one node. The protocol is ready. The network grows when people run nodes.

Deploying a site

Three ways to go live.

All produce the same result — a versioned, replicated deployment with rollback.

01 — DRAG AND DROP

Upload files

Log in to the dashboard, create a site, drag your HTML/CSS/JS into the uploader. Click Deploy. Done.

02 — GIT + AUTO-BUILD

Connect a repo

Point Nexus at a GitHub/GitLab repo. Push to main and the node clones, installs, builds, and deploys. Build cache keeps it fast.

03 — CLI

Use nh deploy

Install the CLI: npm i -g @nexushosting/cli then nh deploy ./dist. Works in CI and GitHub Actions.

Running a node

Five minutes to a working node.

The network grows when people run nodes. A VPS, a Raspberry Pi, a spare machine — the stack runs on 1 GB RAM with LOW_RESOURCE=true. Full guide →

terminal 
# Clone, configure, run
git clone https://github.com/The-No-Hands-company/Nexus-Hosting.git
cd Nexus-Hosting && cp .env.example .env
# Edit .env: set ISSUER_URL, OIDC_CLIENT_ID, COOKIE_SECRET,
# PUBLIC_DOMAIN, and S3 credentials (MinIO included in Compose)
docker compose up -d

What you get: PostgreSQL, Redis, MinIO, the API server, frontend, and Rust proxy — all wired. Caddy handles TLS.

To peer with other nodes, set BOOTSTRAP_URLS in your .env. Your node will handshake, exchange Ed25519 keys, and sync sites automatically.

Honest status

What works. What doesn't. What's next.

Built by one person. Tracked in ROADMAP.md.

// works today

  • Docker Compose — one command to running node
  • Ed25519 federation: handshakes, gossip, site sync
  • Static site hosting with host-header routing
  • Git auto-deploy with build cache
  • Preview deployments (branch → preview URL)
  • Rust proxy: Brotli, LRU cache, Redis invalidation
  • CLI: nh deploy, nh domains, nh teams, nh status
  • Admin: users, moderation, audit log, federation trust
  • Dynamic sites: NLPL, Node.js, Python
  • Custom domains with ACME/TLS automation
  • Email verification, abuse reports, IP bans

// not yet

  • Email+password auth (OIDC required today)
  • Public shared node to try without infra
  • First-run onboarding wizard
  • Malware scanning (hook exists, scanner not bundled)
  • Dynamic site federation across nodes
  • Canonical public seed node list

The biggest friction is auth setup. Running a node requires an OIDC provider (Authentik, Keycloak, Auth0). Adding email+password auth is the next priority.

The network needs nodes.

Federation only works with multiple independent operators. If you run a node — even temporarily on a $6/month VPS — you become part of something that doesn't exist yet: a hosting network nobody controls. Open an issue or start a discussion and I'll help you get set up and peered.
